## Improvement Areas

### P0 — Critical / Foundation

These items fix visible user-facing problems or protect against regressions. They should ship first.

---

#### P0-1: Fix Lighthouse Performance (LCP / TBT)

**Problem:** Mobile Lighthouse performance is `0.43` with LCP `8514ms` and TBT `4048ms`. This is well below the `0.80` / `3500ms` targets already defined in `lighthouserc.json`.

**Root Causes (suspected):**
- `QuestionService.js` is 1,832 lines — even lazy-loaded, the chunk parse + execute cost is high
- MathJax vendor chunk is large; even deferred, it adds to TBT if loaded early in session
- No preconnect/preload hints for the manifest or search-index fetch
- CSS is shipped as one bundle; no critical-CSS extraction

**Safe Improvements:**
1. **Add `<link rel="preload">` for `question-bank-manifest.json`** in `index.html` — zero risk, fetches the manifest in parallel with JS parse
2. **Split `QuestionService.js`** into focused modules:
   - `QuestionNormalizer.js` (normalization, option parsing, UID generation)
   - `QuestionLoader.js` (fetch, cache read/write, init orchestration)
   - `SubjectTaxonomy.js` (subject/subtopic enums, aliases, hierarchy)
   - Keep `QuestionService` as a thin facade that delegates to these
   - **Guard:** All 129 tests must stay green; facade keeps the same public API
3. **Defer non-critical CSS** — extract above-the-fold styles for landing, lazy-load the rest
4. **Add `font-display: swap`** to Google Fonts if any are loaded (currently removed from `index.html` but check Tailwind config)
5. **Tree-shake `react-icons`** — ensure only used icons are bundled (Vite handles this if imports are specific, verify with `npx vite-bundle-visualizer`)

**Risk: LOW** — All changes are additive or internal refactors behind the same public API.

**Acceptance:** Lighthouse mobile performance ≥ 0.65 (interim), LCP ≤ 5000ms

---

#### P0-2: Add Automated Bundle Budget & Lighthouse CI Gate

**Problem:** The startup split was a major win, but there's no automated check to prevent regressions. A single careless import could pull MathJax or the full bank back into the landing chunk.

**Safe Improvements:**
1. **Add a Vite build-size assertion script** (`scripts/qa/validate-bundle-budget.mjs`):
   - Assert landing entry chunk ≤ X KB
   - Assert `vendor-mathjax` chunk is NOT loaded on landing route
   - Assert total initial JS for `/Gate_QA/` ≤ Y KB
2. **Promote `lighthouse:mobile` into CI** (GitHub Actions):
   - Run `npm run build && npm run lighthouse:mobile` on every PR
   - Warn on performance, error on accessibility
3. **Add a network assertion test** (`scripts/qa/validate-landing-network.mjs`):
   - Start Vite preview, fetch the landing HTML, assert no request to `questions-with-answers.json`, `question-search-index.json`, or MathJax

**Risk: LOW** — These are new CI checks, not code changes.

**Acceptance:** CI blocks merges that regress landing payload by >10%.

---

#### P0-3: Accessibility Hardening (0.92 → 0.95+)

**Problem:** Lighthouse accessibility is `0.92`, below the `0.95` error threshold.

**Safe Improvements:**
1. **Audit with `axe-core`** on the landing page, explore page, and solve page
2. **Fix common issues:**
   - Add missing `aria-label` to icon-only buttons (header actions, filter toggle)
   - Ensure color contrast ratios meet WCAG AA on all chip/badge components
   - Add `role="status"` or `aria-live="polite"` to the loading states and exhaustion banner
   - Trap focus inside FilterModal and calculator overlay
   - Add visible focus rings to all interactive elements (some Tailwind defaults may be suppressed)
3. **Add skip-to-content link** on landing

**Risk: LOW** — All changes are additive HTML attributes and CSS.

**Acceptance:** Lighthouse accessibility ≥ 0.95 on landing, explore, and solve pages.

### P1 — High-Value Product Features

These features make the product significantly more useful. Ship after P0 is stable.

---

#### P1-1: Real Question Search

**Problem:** The lightweight search index (`public/question-search-index.json`) is already shipped, but there's no search **UI** on the Explore page. Users can only search via URL params (`?search=`).

**Safe Implementation:**
1. Add a search input to the Explore page filter bar and the FilterSidebar
2. Wire it to the existing `searchQuery` filter state (already in `FilterContext`)
3. Show search results as filtered question cards (existing `QuestionPickerList`)
4. Add a "Clear search" chip to `ActiveFilterChips`
5. Debounce search input (300ms) to avoid excessive filtering
6. The search already uses `useDeferredValue` and stays index-only — no new data fetches

**Risk: LOW** — All plumbing exists. This is a UI-only addition.

**Acceptance:** Users can type a keyword on the Explore page and see matching questions filtered live.

---

#### P1-2: "Report a Bad Question" Flow

**Problem:** Users who find data errors, wrong answers, or broken rendering have no way to report them without going to GitHub directly.

**Safe Implementation:**
1. Add a "Report" icon button to the Solve page question action bar
2. On click, open a new GitHub Issue in a new tab with a prefilled template:
   - `question_uid`
   - Year / Set / Subject
   - Page URL
   - Pre-filled body: "Describe the problem:"
3. Use the GitHub issue URL scheme: `https://github.com/superawat/Gate_QA/issues/new?title=...&body=...`
4. No backend needed — pure URL construction

**Risk: VERY LOW** — Single component addition, no state changes.

**Acceptance:** "Report" button visible on solve page; opens prefilled GitHub issue.

---

#### P1-3: Answer Coverage Visibility & 2026 Answer Backfill

**Problem:** GATE 2026 questions were imported but answers are still `null`. Users see 130 questions with no answers. Additionally, answer coverage gaps across the bank are not surfaced to users.

**Safe Improvements:**
1. **Backfill 2026 answers** via the existing pipeline scripts once GateOverflow publishes them
2. **Show an "Answer unavailable" UI state** in `AnswerPanel` when `answer === null` instead of a blank panel — already partially handled but could be clearer
3. **Surface coverage stats** on the home page: "X% of questions have verified answers"
4. **Add answer coverage to the generated data status** in `docs/generated/DATA_STATUS.md`

**Risk: LOW** — Answer backfill uses existing pipeline; UI changes are cosmetic.

---

#### P1-4: Mock Test Polish & Exit from Beta

**Problem:** Mock test is feature-flagged ON with "Beta Access" label. It needs polish before being promoted as a first-class feature.

**Safe Improvements:**
1. **Review and fix any mock test UX friction:**
   - Timer visibility and countdown behavior
   - Question palette navigation edge cases
   - Results page scoring accuracy
   - Paper gating (ensure only full papers with answers are available)
2. **Add a mock test summary to the home page** (number of mocks attempted, average score)
3. **Remove Beta label** once all edge cases are verified
4. **Add Vitest coverage for mock results scoring logic** (currently tested but may need edge cases)

**Risk: MEDIUM** — Mock test touches several components; changes should be incremental and tested.

---

#### P1-5: Weak-Topic Analytics (Local-Only)

**Problem:** Users solve questions but get no feedback on which subjects/subtopics they're weak in.

**Safe Implementation:**
1. Build a `WeakTopicAnalyzer` utility that reads from `gateqa_progress_v1` (attempt metadata) and `gate_qa_solved_questions`
2. Compute per-subject and per-subtopic:
   - Accuracy rate (correct / attempted)
   - Number attempted vs. available
   - Recent mistake streak
3. Surface as a new "Insights" card on the Home page or a tab in the Explore sidebar
4. All computation is local — no backend, no new storage keys
5. Data stays in localStorage; export via existing JSON export

**Risk: LOW** — Read-only analysis of existing local data; no writes to new keys.

**Acceptance:** Users see a breakdown of their weakest subjects with accuracy percentages.

### P2 — Quality of Life & Polish

These improve developer experience and long-term maintainability.

---

#### P2-1: E2E Browser Test Suite

**Problem:** No browser E2E tests exist. Deep-link, filter, and navigation regressions rely entirely on manual QA and unit tests with mocked DOM.

**Safe Implementation:**
1. Add Playwright as a dev dependency
2. Create a small E2E suite covering:
   - Landing page loads without errors
   - Random practice navigates to a solve page
   - Filter by subject shows matching questions
   - Deep-link `?question=<uid>` opens the correct question
   - Share button copies a valid URL
   - Search input filters results
3. Run E2E in CI after build (separate job from unit tests)
4. Use the same `npm run serve` preview server

**Risk: LOW** — E2E tests are additive and don't modify source code.

---

#### P2-2: Dark Mode

**Problem:** The app currently has a light-only theme. Dark mode is a common user expectation for study tools used at night.

**Safe Implementation:**
1. Extend CSS custom properties with a `[data-theme="dark"]` set
2. Add a theme toggle in the header
3. Persist preference in localStorage under `gate_qa_theme`
4. Use `prefers-color-scheme: dark` as default when no preference is set
5. Scope changes to `index.css` and the theme toggle component — no component logic changes needed if CSS variables are used consistently

**Risk: MEDIUM** — Requires auditing all hardcoded color values in components and MockTest.css (25KB). Should be done as a dedicated pass.

---

#### P2-3: PWA / Offline Support (Phase 1)

**Problem:** Users on trains, in exam halls, or with poor connectivity lose access entirely.

**Safe Implementation — Shell-First:**
1. Add a Vite PWA plugin or manual service worker
2. Cache strategy:
   - **Precache:** app shell (HTML, JS, CSS), landing manifest
   - **Runtime cache:** search index, active detail shard, answer lookups
   - **Skip:** full question bank (too large for automatic cache)
3. Add an install prompt on supported browsers
4. Add an offline fallback page
5. Keep solved/bookmarked progress in localStorage (already works offline)

**Risk: MEDIUM** — Service worker bugs can break the app for users until the cache expires. Needs careful testing and a cache-bust strategy.

---

#### P2-4: QuestionService Refactor

**Problem:** `QuestionService.js` is 1,832 lines with static methods handling normalization, fetching, caching, taxonomy, and option parsing. This is the single largest source of complexity.

**Safe Approach:**
1. Extract into 4-5 focused modules (see P0-1 above for the split plan)
2. Keep `QuestionService` as a thin static facade — same public API, delegating internally
3. All 129 existing tests continue to import and test `QuestionService`
4. New modules get their own test files
5. Do this refactor AFTER P0-1 and P0-2 are landed so bundle budget tests catch regressions

**Risk: MEDIUM** — Internal refactor but touches the most critical service. Must be done with full test coverage verification at each step.

---

#### P2-5: Improve Mobile UX

**Problem:** The app is responsive but the mobile experience has friction points.

**Safe Improvements:**
1. **Swipe gestures** on the Solve page (swipe left = next question, swipe right = previous)
2. **Bottom navigation bar on mobile** (Home, Explore, Mock, Progress)
3. **Collapsible answer panel** on mobile — currently can push content far down
4. **Larger touch targets** for filter chips and action buttons (minimum 44px, per WCAG)
5. **Pull-to-refresh** on Explore page

**Risk: LOW-MEDIUM** — Individual changes are small, but collectively affect layout. Test each independently.

---

#### P2-6: SEO & Social Sharing

**Problem:** As a GitHub Pages SPA, the app is essentially invisible to search engines.

**Safe Improvements:**
1. Add proper `<title>` and `<meta description>` tags in `index.html`
2. Generate a `robots.txt` and `sitemap.xml` during build (list year/subject pages)
3. Add Open Graph meta tags for social sharing
4. Add structured data (JSON-LD) for the question bank as an educational resource

**Risk: VERY LOW** — Build-time additions only.

## Execution Order

The recommended execution sequence, designed so each phase lands independently:

```
Phase A — Foundation Guards (1-2 weeks)
├── P0-1  Lighthouse perf quick-wins (preload, font-display, tree-shake)
├── P0-2  Bundle budget + Lighthouse CI gate
└── P0-3  Accessibility hardening

Phase B — Core Product Features (2-3 weeks)
├── P1-1  Search UI on Explore page
├── P1-2  "Report bad question" flow
└── P1-3  Answer coverage visibility + 2026 backfill

Phase C — Mock & Analytics (2-3 weeks)
├── P1-4  Mock test polish → exit Beta
└── P1-5  Weak-topic analytics

Phase D — Developer & UX Quality (3-4 weeks)
├── P2-1  E2E test suite (Playwright)
├── P2-4  QuestionService refactor
├── P2-5  Mobile UX improvements
└── P2-6  SEO & social sharing

Phase E — Advanced Polish (4-6 weeks)
├── P2-2  Dark mode
└── P2-3  PWA / offline support (Phase 1)
```

Each phase can ship to production independently. No phase depends on unfinished work from a later phase.

## Notes

- **Do not upgrade React to 19** until the ecosystem is stable and all dependencies support it.
- **Do not switch from Tailwind to vanilla CSS** — the codebase is already deeply Tailwind. Migrating would be a massive risk with zero user benefit.
- **Do not add TypeScript** as a migration project. If new files are written, they can be `.ts/.tsx`, but do not convert existing files.
- **Keep GitHub Pages** until a concrete need for server-side features (auth, SSR, edge functions) is identified.
- **Keep the app static-first and local-first** — this is a core architectural principle, not a limitation.
